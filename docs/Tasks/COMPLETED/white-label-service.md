## Executive Summary

This document refines the original White Label Service architecture after thorough validation against the current codebase. The key finding is that **significant theme infrastructure already exists** in IdentityService and the frontend. The strategy must account for this existing work and provide a clear migration path rather than building from scratch.

## Revised Architecture Decision

### Original Plan: Dedicated White Label Service (Option B)

The original architecture doc recommended a standalone `WhiteLabelService`. After auditing the codebase, **I am revising this decision**.

### Revised Decision: Hybrid Approach -- Extend IdentityService + Leverage ContentService

**Rationale**:

1. **Theme configuration is already in IdentityService** -- with Redis caching, ETag support, and production-tested endpoints. Moving this to a new service gains nothing and requires a data migration.

2. **Asset storage is already solved by ContentService** -- the existing SeaweedFS/S3 infrastructure with presigned upload URLs, bucket management, and CDN-compatible paths is exactly what White Label asset management needs. Creating a separate blob storage stack would be wasteful.

3. **Custom domains are a cross-cutting concern** -- DNS verification and SSL provisioning happen at the infrastructure/ingress layer, not at the application service layer. This is better handled by infrastructure configuration with a thin API layer in IdentityService (since domains map to tenants).

4. **Email templates belong in NotificationService** -- the service that sends emails should own the templates. NotificationService already has RabbitMQ consumers and SMTP integration.

5. **Feature flags are a separate concern** -- they deserve their own thin abstraction, possibly a NuGet package, not a dedicated microservice. In the short term, the existing Tenant entity in IdentityService can hold feature flag configuration.

### Architecture Overview

```
                        Frontend (BaseClient)
                              |
              +---------------+---------------+
              |               |               |
     IdentityService   ContentService   NotificationService
     (Theme Config,    (Asset Storage,  (Email Templates,
      Custom Domains,   Logos, Favicons) Template Rendering)
      Feature Flags)
              |               |               |
        IdentityDB      ContentDB +       NotificationDB
                         SeaweedFS
              |               |               |
              +-------+-------+-------+-------+
                      |               |
                    Redis          RabbitMQ
                  (Cache)        (Events)
```

### What Goes Where

| Capability | Service | Justification |
|-----------|---------|---------------|
| Theme configuration (colors, typography, presets) | **IdentityService** | Already exists and working |
| Theme preset management | **IdentityService** | Already exists with 5 presets |
| Feature flags per tenant | **IdentityService** | Tenant is the aggregate root here |
| Custom domain registration & verification | **IdentityService** | Domains map to tenants |
| Logo/Favicon/Asset upload & storage | **ContentService** | S3 storage already operational |
| Asset CDN URL generation | **ContentService** | Presigned URLs already implemented |
| Email template management | **NotificationService** | Owns email sending pipeline |
| Email template rendering | **NotificationService** | Has SMTP/Mailpit integration |
| Branding asset references (logo content ID) | **IdentityService** | Stored as content IDs in ThemeConfigJson |

### Why NOT a Dedicated Service

| Concern | Counter-argument |
|---------|------------------|
| "SRP violation" | Theme config is a tenant attribute -- IdentityService owns tenants. Assets go to ContentService. Emails go to NotificationService. Each service handles its natural responsibility. |
| "Scaling independently" | Theme configs are tiny JSON blobs cached in Redis. There is no independent scaling need. |
| "New service is cleaner" | Adding a 7th microservice with its own DB, Docker image, Tilt resources, and deployment pipeline creates more operational complexity than the problem warrants. |
| "Feature bloat" | The additions to IdentityService are 3-4 new entities and 10-15 endpoints -- modest for a service that already has Tenants, Users, Auth, Privacy, and Sessions. |

#### Existing Entity: Tenant (No Schema Changes)

The `ThemeConfigJson` field already stores the full theme configuration. The `LogoUrl` and `PrimaryColor` fields are legacy but can remain for backward compatibility. The `ThemeConfigJson` includes `LogoContentId` and `FaviconContentId` which reference content items in ContentService.

### ContentService Extensions

No schema changes needed. The existing `ContentItem` entity already supports:
- Multi-tenant storage (`BaseTenantEntity` with `TenantId`)
- File metadata (`FileName`, `ContentType`, `FileSizeBytes`, `MetadataJson`)
- Blob path tracking (`BlobPath`)
- Status management (`Uploading -> Processing -> Ready`)
- Public/private access (`IsPublic`)

White label assets (logos, favicons) are just content items with a specific category. Add a new `ContentCategory` enum value for `BrandingAsset`.

### Tenant Resolution Strategy

The existing frontend loads the tenant from the JWT token (`tenantId` claim). For pre-login white labeling (custom domain scenario):

1. **Web**: Extract hostname from `window.location.hostname`
2. **Call** `GET /tenants/resolve?domain={hostname}` to get tenant ID
3. **Load** theme config from `GET /tenants/{tenantId}/theme`
4. **Cache** in AsyncStorage with 5-minute TTL
5. **Fallback** to default theme if resolution fails

This matches the strategy in the original doc but uses the existing IdentityService endpoint pattern.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| IdentityService becomes too large | Low | Medium | Feature flag and domain endpoints are self-contained; easy to extract later if needed |
| Theme changes don't propagate to all components | Medium | High | Comprehensive E2E tests with visual regression |
| Custom domain DNS verification is unreliable | Medium | Medium | Retry mechanism with exponential backoff; manual verification override for super admins |
| Image processing adds complexity to ContentService | Low | Low | Keep validation simple (dimensions/size checks only); no server-side resizing in Phase 1 |
| Email template XSS vulnerabilities | Medium | High | Sanitize template HTML; use Liquid templating which auto-escapes by default |
| Cache invalidation races | Low | Medium | Use Redis pub/sub or MassTransit events for cache busting; accept eventual consistency (max 15min stale) |

## Open Questions (To Resolve Before Implementation)

1. **Template engine**: Liquid (DotLiquid/Fluid) vs Handlebars vs Razor? Liquid is recommended -- it is safe by default (no code execution), widely used, and has good .NET support via the `Fluid` library.

2. **Custom fonts**: Should Phase 1 support custom font uploads? **Recommendation**: No. Use system fonts and Google Fonts only in Phase 1. Custom font hosting adds CDN complexity.

3. **Domain SSL**: Let's Encrypt automatic provisioning vs manual? **Recommendation**: Start with manual/infrastructure-level SSL. Automatic Let's Encrypt requires a reverse proxy integration (Caddy/Traefik) that is a separate infrastructure task.

4. **Theme versioning**: Should theme changes create an audit trail? **Recommendation**: Not in Phase 1. Add `ThemeVersion` field to Tenant entity later if rollback is needed.

5. **Feature flag SDK**: Should we create a client-side SDK package? **Recommendation**: Not yet. The simple `useFeatureFlag()` hook is sufficient. Consider LaunchDarkly or a dedicated flag service if flag complexity grows.
