# Wire Onboarding Hooks to Server-Side State

## Status: COMPLETED

## Problem Statement
The welcome wizard and setup checklist hooks (`useWelcomeWizard`, `useSetupChecklist`) currently
persist their completion/dismissal state only in localStorage. The Identity service now exposes
`wizardCompleted` and `checklistDismissed` fields on the user preferences endpoint
(`GET/PUT /me/preferences`). We need to wire the frontend hooks to use server-side persistence
as the authoritative source, with localStorage as a fast-read cache.

## Implementation Summary

### Step 1: Regenerate hooks
- Updated swagger spec with `wizardCompleted` and `checklistDismissed` boolean fields
- Ran `generate:hooks:skip-download` to regenerate TypeScript types

### Step 2: Updated `useWelcomeWizard.ts`
- Added `PreferencesInput` interface (shared between wizard and checklist hooks)
- Hook now accepts `preferences: PreferencesInput` as second parameter
- Derives `serverWizardDone` from `preferences.data?.wizardCompleted`
- Keeps localStorage as fast-path (`localWizardDone`)
- `isWizardVisible` prevents flash: false while `preferences.isLoading`
- `handleComplete`: writes localStorage + calls `preferences.updatePreferences`
- Refactored to `useWizardFormState()` and `buildWizardResult()` to stay under 30-line limit

### Step 3: Updated `useSetupChecklist.ts`
- Hook now accepts `preferences: PreferencesInput` as second parameter
- Derives dismissed from `localDismissed || serverDismissed`
- `handleDismiss`: writes localStorage + calls `preferences.updatePreferences`
- QR/share flags remain localStorage-only (transient session actions)

### Step 4: Updated `DashboardPage.tsx`
- Added `usePreferencesInput()` local hook calling GET and PUT preferences once
- Returns `PreferencesInput` object passed to both hooks
- Single network request serves both wizard and checklist

### Step 5: Updated tests
- `useWelcomeWizard.test.ts`: 5 new server-state tests
- `useSetupChecklist.test.ts`: 4 new server-state tests
