# ESLint Component Reusability Rules

## Status: COMPLETED

## Problem Statement
Shared components in `src/components/` should not import from product-specific modules. This prevents tight coupling and ensures reusability. Additionally, we need to warn when product modules create components that duplicate existing shared components.

## Rules Created

### Rule 1: `no-product-imports-in-shared` (WARN)
Bans product-specific imports inside `src/components/`.

**Severity**: Set to `warn` (not `error`) because 24 pre-existing violations exist in the codebase. These should be migrated over time by moving product-specific components out of `src/components/` into their respective feature module packages.

**Banned import patterns** (when importing file is under `src/components/`):
- `@baseclient/*-module` (except `identity-module`, which is shared infra)
- `server/autoGeneratedHooks/{onlinemenu,questioner,content,notification}/`
- `src/features/` or `src/modules/`

**Allowed**:
- `@baseclient/core`, `@baseclient/identity-client`, `@baseclient/identity-module`
- External packages (react, react-native, etc.)
- Relative imports within `src/components/`

**Pre-existing violations** (24 total): Primarily in `QuestionerTemplates/`, `OnlineMenus/`, `Dashboard/`, `Settings/`, and `Buttons/ExportButtons/` components that import from product-specific auto-generated hooks.

### Rule 2: `enforce-component-accessibility` -- SKIPPED
Already covered by existing `react-native-a11y` plugin rules:
- `has-accessibility-hint` (warn)
- `has-accessibility-props` (error)
- `has-valid-accessibility-descriptors` (error)

### Rule 3: `no-duplicate-shared-patterns` (WARN)
Warns when product module directories (`packages/*-module/src/`, `src/features/`) create `.tsx` components with the same basename as existing shared components in `src/components/`.

Uses filesystem scanning with caching for performance.
